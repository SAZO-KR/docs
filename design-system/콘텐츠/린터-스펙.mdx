SAZO(사줘)의 UX Writing 가이드라인을 자동 검사하는 린터 시스템의 구현 가이드다. [문법 및 스타일](/design-system/콘텐츠/문법-및-스타일)에 정의된 권장/비권장 규칙을 프로그래매틱하게 체크하며, [보이스 및 톤](/design-system/콘텐츠/보이스-및-톤)과 [UI 패턴](/design-system/콘텐츠/UI-패턴)의 원칙도 함께 반영한다.

## 왜 만드는가
UX Writing 가이드를 아무리 잘 만들어도 **사람이 매번 읽고 판단해야 하면 지켜지지 않는다.** 자동화된 피드백 루프가 없으면 가이드는 점점 형해화된다. "이 텍스트가 사용자에게 얼마나 어려운가"를 정량적으로 보여주는 도구는 실제 가치가 있다.

원래 아이디어는 "단어의 인지 부하를 측정하는 LLM → 시각화 서비스 → 개발 워크플로우 통합"이었다. 하지만 이 순서는 위험하다. 핵심 위험 두 가지를 먼저 짚는다.

### "인지 부하"의 정의 문제
"인지 부하"는 단일 숫자가 아니다. 최소 다섯 가지 차원이 있고, 각각 측정 가능성이 다르다:

| 차원 | 측정 가능성 | 예시 |
|---|---|---|
| 어휘 난이도 | 높음 (코퍼스 빈도) | "통관" vs "세관 검사" |
| 문장 구조 복잡도 | 높음 (구문 분석) | 종속절 깊이, 문장 길이 |
| 도메인 전문성 | 중간 (용어 사전) | "3D Secure"는 일반인에겐 어렵고 개발자에겐 쉬움 |
| 맥락 의존 모호성 | 낮음 (판단 필요) | "처리"가 결제 처리인지 반품 처리인지 |
| 문화적 배경 | 낮음 (주관적) | 일본 거주 한국인 vs 한국 거주 한국인 |

상위 2개는 LLM 없이도 측정 가능하다. 하위 3개는 LLM이 도움이 되지만, 정답이 없어서 검증이 어렵다. **검증할 수 없는 측정은 측정이 아니라 의견이다.** LLM이 "이 문장은 인지 부하 7.2"라고 했을 때, 그게 맞는지 틀린지 어떻게 아는가?

### LLM을 "학습"시키지 마라
가장 큰 함정은 "인지 부하 측정 전용 LLM을 만들어야 한다"고 생각하는 것이다.

파인튜닝하려면 라벨링된 데이터가 필요한데, "이 문장의 인지 부하는 X"라는 골드 라벨을 대량으로 만드는 것 자체가 연구 프로젝트급이다. 사줘 규모에서 이걸 하면 본업을 못 한다. 대신 기존 LLM에 프롬프트로 판단을 요청하는 LLM-as-Judge 방식을 쓰되, 그것도 레이어 1(규칙 기반)이 80%의 가치를 만든 뒤에 나머지 20%를 채우는 용도로만 쓴다.

## 시작 전 핵심 질문
프로젝트 착수 전에 다음 네 가지에 답해야 한다:

1. **누가 쓰는가?** — 라이터, 디자이너, 개발자, PM 중 주 사용자가 누구인지에 따라 인터페이스와 피드백 방식이 달라진다. 라이터에게는 에디터 플러그인이, 개발자에게는 CI 게이트가 자연스럽다.
2. **어떤 시점에 쓰는가?** — 작성 중(실시간 피드백), 리뷰 시(PR 코멘트), 배포 전(CI 게이트) 각각 요구하는 응답 속도와 피드백 형태가 다르다.
3. **"좋다/나쁘다"의 기준은 무엇인가?** — 점수 임계값을 누가, 어떤 근거로 정하는가. 이게 없으면 도구를 만들어도 아무도 신뢰하지 않는다.
4. **사줘 전용인가, 범용인가?** — 사줘 전용이면 용어 사전과 톤 규칙을 하드코딩할 수 있다. 범용 서비스를 만들려는 거면 완전히 다른 규모의 프로젝트다.

## 전체 아키텍처
### 레이어 구분
**레이어 1: 규칙 기반 (LLM 불필요)**
- 결정적(deterministic) — 같은 입력에 항상 같은 결과. LLM처럼 "오늘은 7.2, 내일은 6.8" 이런 일이 없다.
- 정규표현식, 사전 룩업, 형태소 분석기(Kiwi/Mecab-ko)로 처리
- 빠르고 CI에 넣기 적합
- 전체 가치의 약 80%를 차지

**레이어 2: LLM-as-Judge (기존 모델 프롬프트)**
- "이 문장을 이커머스에 익숙하지 않은 사용자가 읽을 때 어떤 부분에서 멈출 가능성이 있는가?"
- 대안 문장 제안
- 맥락 의존적 모호성 플래그
- 레이어 1로 잡히지 않는 나머지 20%를 담당

CLI 도구 또는 API로 구현한다. 입력: 텍스트 → 출력: 위반 목록 + 심각도. 가이드에 이미 "권장/비권장"이 명확하니까 규칙 추출이 바로 가능하다. **이 단계에서 팀이 실제로 쓰는지 검증하는 것이 핵심이다.**

#### 기본 도구: 형태소 분석기
LLM과 형태소 분석기는 전혀 다른 것이다:
- **LLM** : 확률적으로 "다음 토큰"을 생성하는 거대 모델 (수십~수천억 파라미터)
- **형태소 분석기** : 문장을 문법 단위로 쪼개는 규칙+사전 기반 도구 (수 MB)

한국어 형태소 분석기 후보:

| 도구 | 특징 |
|---|---|
| Mecab-ko | 가장 빠름, C++ 기반, 사전 확장 가능 |
| Komoran | 자바 기반, 오타에 강함 |
| Kiwi | 최신, 속도/정확도 균형, Python 네이티브 |

Kiwi 사용 예시:

```python
from kiwipiepy import Kiwi
kiwi = Kiwi()
tokens = kiwi.tokenize("통관 절차가 완료되었습니다")

# 통관/NNG  절차/NNG  가/JKS  완료/NNG  되/XSV  었/EP  습니다/EF
```

각 토큰에 **품사 태그** 가 붙는다. NNG는 일반명사, XSV는 동사 파생 접미사, EF는 종결어미. 이 정보만으로 레이어 1의 거의 모든 것을 할 수 있다.

#### B. 사전 룩업 (매칭 테이블)
가이드에 이미 "이것 대신 저것을 써라"가 테이블로 정리되어 있다. 그대로 사전화하면 된다. 문자열 `in` 연산이므로 해시맵이면 O(1).

```python

# 외래어 → 한국어 대체어
term_replacements = {
    "카트": "장바구니",
    "위시리스트": "찜하기 / 찜한 상품",
    "에러": "오류",
    "Favorite": "찜하기",
    "Cart": "장바구니",
    "Error": "오류",
}

# 전문 용어 → 사용자 대면 표현
jargon_replacements = {
    "캐시 초기화": "저장된 데이터 삭제",
    "세션 만료": "로그인 시간 초과",
    "토큰": "인증 정보",
    "동기화": "정보 업데이트",
}

# 포용적 표현
inclusive_replacements = {
    "회원님": "이용자 / 고객",
    "남녀 모두": "모든 고객",
    "배송기사": "배송 담당자",
    "판매사원": "판매원",
}

def check_terms(text, dictionary, rule_ref):
    findings = []
    for banned, suggested in dictionary.items():
        if banned in text:
            findings.append({
                "term": banned,
                "suggestion": suggested,
                "ref": rule_ref
            })
    return findings
```

#### C. 형태소 분석기 활용
띄어쓰기, 종결 어미, 수동태, 외래어 비율 — 형태소 분석기의 품사 태그를 활용해야 하는 규칙들이다. 여전히 LLM은 아니다.

**보조 용언 띄어쓰기**

가이드에서 가장 규칙이 명확한 부분이다. "보조 용언은 띄어 쓴다"가 원칙이고, 보조 용언 목록이 한정되어 있다. 형태소 분석기가 `VV`(동사), `VX`(보조용언)를 태깅해주기 때문에 가능하다.

`"입력해주세요"` → `입력/NNG + 하/XSV + 아/EC + 주/VX + 세요/EF` → VX 앞에 공백 없음 → 위반.

```python
auxiliary_verbs = ["주다", "보다", "드리다", "있다", "없다", "하다"]

def check_auxiliary_spacing(tokens):
    """연속된 두 용언이 공백 없이 붙어 있는지 체크"""
    findings = []
    for i in range(len(tokens) - 1):
        curr = tokens[i]
        next_t = tokens[i + 1]
        # 본용언(VV/VA) 바로 뒤에 보조용언(VX)이 오는데
        # 원문에서 공백이 없었다면 → 위반
        if curr.tag in ("VV", "VA") and next_t.tag == "VX":
            if no_space_between(curr, next_t):  # 원문 위치 기반 판단
                findings.append({
                    "position": curr.start,
                    "found": f"{curr.form}{next_t.form}",
                    "suggestion": f"{curr.form} {next_t.form}",
                    "message": "보조 용언은 띄어 쓴다"
                })
    return findings
```

**종결 어미**

가이드에 허용/금지 어미가 명확하다. 종결어미는 형태소 분석기가 `EF` 태그로 정확히 찍어준다.

```python
banned_endings = {
    "하십시오": "지나치게 권위적",
    "해요": "격식체 통일 원칙에 어긋남",
    "어요": "격식체 통일 원칙에 어긋남",
    "에요": "격식체 통일 원칙에 어긋남",
    "함": "비격식적 (반말 명사형)",
    "됨": "비격식적 (반말 명사형)",
}

def check_ending(tokens):
    last_token = tokens[-1]  # 문장 마지막 토큰
    if last_token.tag == "EF":  # 종결어미
        for banned, reason in banned_endings.items():
            if last_token.form == banned:
                return {"found": banned, "reason": reason}
    return None
```

**수동태 (번역투)**

한국어 피동 표현은 패턴이 정해져 있다:

```python
passive_suffixes = {"이", "히", "리", "기"}     # 피동 접미사
passive_verbs = {"되다", "받다", "당하다"}        # 피동 동사

def check_passive(tokens):
    for i, t in enumerate(tokens):
        # "~에 의해 ~되다" 패턴 (번역투)
        if t.form == "의해" or (t.tag == "XSV" and t.form == "되"):
            # "완료되다"는 허용, "~에 의해 처리되다"는 경고
            # 앞에 "에 의해"가 있으면 번역투 의심
            if i >= 2 and tokens[i-1].form == "의해":
                return {"message": "번역투 수동태입니다. 능동태로 바꿔 쓰세요"}
    return None
```

핵심: "결제가 완료되었습니다"까지 전부 잡으면 false positive가 너무 많다. 가이드도 이걸 전면 금지하지 않고 "~에 의해" 번역투만 문제 삼고 있으므로, **패턴의 범위를 가이드의 비권장 범위와 일치** 시켜야 한다.

**외래어/전문용어 비율**

형태소 분석기가 외국어를 `SL` 태그로 구분해준다:

```python
for token in kiwi.tokenize("3D Secure 인증이 필요합니다"):
    print(token.form, token.tag)

# 3/SN  D/SL  Secure/SL  인증/NNG  이/JKS  필요/NNG  합니다/EF
# SL = 외국어, SN = 숫자
```

```python
def foreign_ratio(tokens):
    content_tokens = [t for t in tokens if t.tag.startswith(("N", "V", "SL"))]
    foreign = [t for t in content_tokens if t.tag == "SL"]
    return len(foreign) / len(content_tokens) if content_tokens else 0
```

전문용어는 사줘 도메인 용어 사전을 별도로 만들어서 매칭한다. 가이드의 [두문자어](/design-system/콘텐츠/문법-및-스타일#두문자어)와 [외래어 표기](/design-system/콘텐츠/문법-및-스타일#외래어-표기) 섹션에 이미 용어 기준이 있으니, 그걸 사전화하면 된다.

#### D. 휴리스틱 (임계값 기반)
**문장 길이**

가이드에 이미 숫자가 있다: "권장 15~60자, 최대 120자".

```python
def check_verbosity(text):
    findings = []
    sentences = split_sentences(text)
    for s in sentences:
        length = len(s)
        if length > 120:
            findings.append({
                "severity": "error",
                "message": f"문장이 {length}자입니다. 120자를 초과하면 분리하세요"
            })
        elif length > 60:
            findings.append({
                "severity": "warning",
                "message": f"문장이 {length}자입니다. 60자 이하를 권장합니다"
            })
    return findings
```

**어휘 난이도**

국립국어원이 공개한 현대국어 사용빈도 조사 데이터를 활용한다. 약 7만 어휘에 빈도 등급이 매겨져 있고, 한국어 학습용 어휘 목록은 등급(A/B/C)까지 분류되어 있다.

```python

# 빈도 사전 (실제로는 CSV/DB에서 로드)
freq_dict = {
    "결제": {"rank": 3200, "grade": "B"},
    "통관": {"rank": 28000, "grade": "C"},
    "주문": {"rank": 1800, "grade": "A"},
}

def word_difficulty(token):
    entry = freq_dict.get(token.form)
    if entry is None:
        return "unknown"  # 사전에 없으면 높은 난이도로 간주
    if entry["rank"] < 5000:
        return "easy"
    if entry["rank"] < 15000:
        return "medium"
    return "hard"
```

이것은 사전 검색이다. O(1) 해시맵 룩업이라 수만 문장을 초 단위로 처리한다. "통관"이 28,000위라서 hard로 나오면 → "세관 검사"(세관: 8,000위 + 검사: 2,100위)를 대안으로 제시할 수 있는 것도 같은 사전에서 나온다.

**문장 복잡도**

의존 구문 분석(dependency parsing)으로 종속절 깊이를 측정한다. 이것도 소형 모델(수십 MB)이지 LLM이 아니다.

```
단순: "결제가 완료되었습니다."

복잡: "고객님이 선택하신 상품의 결제가 정상적으로 완료되었습니다."
```

텍스트를 입력하면 어떤 부분이 어려운지 시각적으로 보여주는 도구다.

- **히트맵 오버레이** : 레이어 1 점수 기반으로 텍스트 위에 색상 렌더링. 어휘 난이도가 높은 단어는 붉게, 문장 구조가 복잡한 부분은 진하게 표시.

<Accordion title="Phase 3: LLM 레이어 (Phase 2 검증 후)">

<Note>

진입 조건: Phase 2의 시각화가 검증되고, 레이어 1만으로 커버되지 않는 케이스가 충분히 수집된 상태. "이런 건 왜 안 잡아주지?"라는 피드백이 쌓여야 LLM 도입이 정당화된다.

</Note>

규칙으로 잡히지 않는 맥락적 문제를 LLM으로 보완한다. **파인튜닝하지 않는다.** 기존 LLM에 프롬프트로 판단을 요청하는 LLM-as-Judge 방식을 쓴다.

레이어 1으로 안 되는 것들:

| 규칙 | 왜 규칙 기반으로 안 되는가 |
|---|---|
| 관점 일관성 (1인칭/2인칭 혼용) | "한 화면" 단위를 린터가 모른다. 입력이 화면 단위가 아니면 판단 불가 |
| 느낌표 맥락 판단 | "이게 프로모션인가 오류 메시지인가"는 UI 맥락 의존 |
| "해보세요" 예외 | 격려 문맥인지 일반 문맥인지 의미 판단 필요 |
| 장황함의 질적 판단 | 글자 수는 재지만 "불필요한 수식"은 의미 판단 |
| 직접적인 문장 | "주체가 명확한가"는 구문만으로 부족, 의미 레벨 판단 필요 |

프롬프트 설계 방향:

```
당신은 SAZO(사줘)의 UX Writing 전문 리뷰어입니다.
다음 텍스트를 검토하고, 아래 기준에 해당하는 문제가 있으면 지적하세요:

1. 한 화면 내에서 1인칭("내")과 2인칭("고객님")이 혼용되고 있는가?
2. 오류/경고 맥락에서 느낌표가 사용되었는가?
3. 불필요한 수식이나 반복으로 핵심 메시지가 희석되는가?
4. 행위의 주체가 불분명한 피동 표현이 있는가?

텍스트: {input_text}
UI 맥락: {context}  (예: 에러 모달, 프로모션 배너, 결제 확인)
```

LLM 레이어의 결과는 레이어 1과 다르게 **제안(suggestion)** 으로 표시한다. 레이어 1은 "위반(violation)"이지만, LLM 판단은 확률적이므로 "검토해 보세요(review)" 수준으로 구분해야 신뢰를 유지할 수 있다.

</Accordion>

<Accordion title="TSX 인라인 텍스트 — AST 파싱">

i18n 없이 TSX에 텍스트가 직접 들어가 있으면, 코드와 사용자 텍스트를 AST(구문 트리) 레벨에서 분리해야 한다. 한 파일 안에서도 린트 대상과 비대상이 섞여 있다:

```tsx
export function OrderComplete() {
  // ❌ 린트 비대상: 주석
  // 결제 완료 후 리다이렉트 처리
  
  const API_URL = "/api/orders";  // ❌ 비대상: 변수값
  
  console.log("payment completed"); // ❌ 비대상: 로그
  
  return (
    <div>
      <h1>결제가 완료되었습니다</h1>     {/* ✅ 대상: JSX 텍스트 노드 */}
      <p>배송은 3~5일 소요됩니다</p>     {/* ✅ 대상 */}
      <Button label="장바구니에 담기" />  {/* ✅ 대상: 특정 prop의 문자열 */}
      <input 
        placeholder="검색어를 입력해 주세요"  // ✅ 대상
        name="search"                        // ❌ 비대상: 기술 속성
        data-testid="search-input"           // ❌ 비대상
      />
      <img alt="상품 이미지" />              {/* ✅ 대상: 접근성 텍스트 */}
    </div>
  );
}
```

이걸 구분하는 건 TypeScript/Babel AST 파서로 가능하다:

<Frame>

```

  │
  ├─ TypeScript Parser (ts-morph 또는 @babel/parser)
  │
  ▼
```

ESLint 플러그인으로 구현하면 기존 린트 파이프라인에 바로 붙는다:

```js
// eslint-plugin-sazo-writing/rules/no-currency-symbol.js
module.exports = {
  create(context) {
    return {
      // JSX 텍스트 노드
      JSXText(node) {
        check(node.value, node, context);
      },
      // 특정 prop의 문자열 값
      JSXAttribute(node) {
        const uiProps = ["label", "title", "placeholder", 
                         "alt", "aria-label", "description"];
        if (uiProps.includes(node.name.name) && 
            node.value?.type === "Literal") {
          check(node.value.value, node, context);
        }
      }
    };
    
    function check(text, node, context) {
      if (/[¥₩]\s?\d/.test(text)) {
        context.report({
          node,
          message: "통화 기호 대신 '엔', '원'을 사용하세요"
        });
      }
    }
  }
};
```
