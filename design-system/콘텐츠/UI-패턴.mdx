## 행동 유도형 UI

<AccordionGroup>

<Accordion title="버튼 및 CTA">

**역할과 목표**
사용자의 다음 행동을 명확하게 안내하여, 망설임 없이 클릭할 수 있게 한다.

**글자 수 가이드**
- 최소: 2자 (예: "확인")
- 권장: 4~8자
- 최대: 12자 (그 이상은 설명 텍스트로 분리)
- **접근성 근거:** 최대 12자 제한은 시각적 간결함뿐 아니라 터치 타겟 접근성(최소 9mm 간격)을 보장하기 위함입니다. 버튼 텍스트가 길어지면 인접 버튼과의 간격이 좁아져 터치 오작동(Fat Finger Error) 위험이 높아집니다.

**톤 가이드**
- 간결하고 행동 중심. 격식체 종결 어미 대신 **명사형 또는 ~하기/~보기** 패턴 사용.
- 부드럽되 명확하게. 강요하는 어감을 피한다.
- **줄바꿈 방지:** 버튼은 1줄 표시가 필수인 컴포넌트이므로, 띄어쓰기를 최소화하여 시각적 덩어리(Chunk)로 인식되도록 한다. 복합 명사는 붙여쓰기를 허용한다. (예: "구매대행 신청하기"가 "구매 대행 신청하기"보다 줄바꿈에 강하다.) 개발 시 `white-space: nowrap` 처리를 권장한다.
- **접근성(aria-label):** 시각적 UI에서 공간 제약으로 붙여쓰기를 적용한 경우, 해당 요소의 `aria-label`에는 맞춤법대로 띄어 쓴다. (예: 버튼 텍스트 "구매대행" → aria-label "구매 대행") 이렇게 하면 시각적 줄바꿈 방지와 스크린 리더 호환성을 동시에 달성할 수 있다.

**문장 구조 패턴**
- 권장 형식 1: **[동사]하기** — "수정하기", "결제하기", "복사하기"
- 권장 형식 2: **[목적어] [동사]하기** — "상품 목록 보기", "홈으로 돌아가기"
- 권장 형식 3: **[명사]** (보조 버튼) — "확인", "취소", "삭제"
- 기본(Primary) CTA는 ~하기/~보기 행위형, 보조(Secondary) 버튼은 명사형.

**Do/Don't 예시**

| Do | Don't | 이유 |
| --- | --- | --- |
| 결제로 진행하기 | 결제 진행 버튼 클릭 | 버튼 자체가 클릭 대상이므로 "버튼 클릭" 표현은 불필요 |
| 상품 보러 가기 | 상품 목록 이동 | 일상어로 행동을 안내해야 자연스럽다 |
| 홈으로 돌아가기 | 홈 화면으로 리다이렉트 | 기술 용어(리다이렉트)를 피한다 |

**CTA 보조 텍스트**
버튼 아래에 신뢰나 동기를 더하는 짧은 보조 텍스트를 배치할 수 있다. 특히 결제 전환이 핵심인 CTA에서 효과적이다.

| 상황 | 보조 텍스트 예시 |
| --- | --- |
| 결제 불안 해소 | "추가 비용은 없습니다" |
| 진입 장벽 낮추기 | "카드 등록 불필요" |
| 소요 시간 안내 | "1분이면 완료" |
| 되돌림 가능성 | "언제든 취소할 수 있습니다" |

**결과 중심 CTA 원칙**
버튼 텍스트는 **기능이 아닌 사용자가 얻는 결과** 를 강조한다. "다운로드"보다 "알림 바로 받기"가 행동 동기를 높인다.

| 기능 중심 (Don't) | 결과 중심 (Do) |
| --- | --- |
| 앱 다운로드 | 알림 바로 받기 |
| 필터 적용 | 맞춤 상품 보기 |
| 회원가입 | 혜택 받기 |

</Accordion>

<Accordion title="플레이스홀더">

**역할과 목표**
입력 필드에서 사용자가 어떤 정보를 입력해야 하는지 안내한다.

**글자 수 가이드**
- 최소: 4자 (예: "검색어 입력")
- 권장: 8~20자
- 최대: 30자 (모바일에서 잘리지 않도록 제한)

**톤 가이드**
- 부드러운 제안형. ~해 주세요 또는 ~해 보세요 패턴 사용.
- 마침표를 생략한다 (입력 필드 안의 보조 텍스트이므로).
- 느낌표는 검색 등 탐색형 필드에서만 제한적으로 사용한다.

**문장 구조 패턴**
- 권장 형식 1: **[대상]을(를) 입력해 주세요** — "전화번호를 입력해 주세요"
- 권장 형식 2: **[행동 유도 질문형]** — "궁금한 상품을 찾아보세요!"
- 권장 형식 3: **[예시 안내]** — "예) OOO-OOOO"

**Do/Don't 예시**

| Do | Don't | 이유 |
| --- | --- | --- |
| 궁금한 상품을 찾아보세요 | 검색어를 입력하십시오 | 탐색형 필드에서는 친근한 유도가 자연스럽다 |
| 전화번호를 입력해 주세요 | 전화번호 | 행동을 안내해야 사용자가 무엇을 해야 할지 명확히 안다 |
| 예) 010-1234-5678 | ex) 010-1234-5678 | 한국어 UI에서는 "예)"를 사용한다 |

</Accordion>

<Accordion title="링크 텍스트">

**역할과 목표**
본문, 도움말, 약관 안내 등에서 사용자를 다른 페이지나 외부 리소스로 안내한다.

**글자 수 가이드**
- 최소: 2자
- 권장: 4~15자
- 최대: 25자 (문장 전체를 링크로 감싸지 않는다)

**톤 가이드**
- **목적지를 명시한다.** "여기", "이곳", "클릭" 같은 모호한 표현을 금지한다.
- 링크 텍스트만으로 어디로 이동하는지 알 수 있어야 한다. 스크린 리더 사용자는 링크 목록만 따로 탐색하므로, 맥락 없이 "여기"만 읽히면 정보를 얻을 수 없다.
- 외부 링크는 새 탭으로 열리며, "(새 탭)" 또는 아이콘으로 표시한다.

**문장 구조 패턴**
- 권장 형식 1: **[목적지/내용 명사]** — "이용약관", "개인정보 처리방침"
- 권장 형식 2: **[내용] 보기/확인하기** — "배송 정책 확인하기", "자세한 내용 보기"
- 권장 형식 3: **[동사구]** — "고객센터에 문의하기"

**동사 포함 기준: 시각적 어포던스에 따른 판단**
시각적 어포던스(밑줄, 색상, 아이콘 등)와 텍스트 어포던스(동사)는 보완 관계다. 둘의 합이 충분하면 사용자가 "이건 누르는 거구나"를 인지한다. 시각 어포던스가 높으면 동사가 불필요하고, 낮을수록 동사가 필수적이다.

| 컴포넌트 | 시각 어포던스 | 동사 포함 | 예시 |
| --- | --- | --- | --- |
| 밑줄 링크 | 높음 | 불필요 (aria-label로 보완) | `받는 사람 목록` |
| 텍스트 버튼 (색상 구분) | 중간 | 권장 | `목록 보기` |
| 쉐브론(>) 동반 | 중간 | 권장 | `받는 사람 목록` + > |
| 플레인 텍스트 | 없음 | 필수 | `받는 사람 목록 보기` |

- **접근성 보완:** 밑줄 링크에서 동사를 생략할 경우, 스크린 리더는 밑줄을 인지하지 못하므로 `aria-label`에 동사를 포함하여 보완한다.
    - 예: `<a href="..." aria-label="받는 사람 목록 보기">받는 사람 목록</a>`
    - 화면에는 간결하게, 스크린 리더에는 명확하게.
- 시각 어포던스가 불확실한 경우(디자인 미확정, 다크모드에서 색상 구분 약화 등), 동사를 포함하는 쪽이 안전하다.

**Do/Don't 예시**

| Do | Don't | 이유 |
| --- | --- | --- |
| 배송 정책 확인하기 | 여기를 클릭하세요 | 목적지가 명확해야 한다 |
| 이용약관 (밑줄 링크) | 이 링크 | 링크 텍스트만으로 내용을 알 수 있어야 한다 |
| 고객센터에 문의하기 | 자세한 내용은 여기 | 스크린 리더가 "여기"만 읽으면 정보가 없다 |
| 받는 사람 목록 (밑줄) | 받는 사람 목록 (플레인 텍스트) | 시각 어포던스 없이 동사도 없으면 클릭 가능성을 인지할 수 없다 |

</Accordion>

<Accordion title="대체 텍스트(alt)">

**역할과 목표**
이미지, 아이콘 등 비텍스트 콘텐츠의 의미를 스크린리더 사용자에게 전달한다. 시각 정보에 접근할 수 없는 사용자도 동등한 맥락을 얻을 수 있어야 한다.

**작성 규칙**
- **"이미지", "그림", "사진 of" 등을 포함하지 않는다.** 스크린리더가 이미 `<img>` 요소를 "이미지"로 읽어 주기 때문에 중복된다.
- **"버튼", "링크" 등 역할명을 포함하지 않는다.** HTML 역할(role)이 이미 명시되어 있다.
- **인접 텍스트와 동일한 내용을 반복하지 않는다.** 캡션이나 라벨이 이미 같은 정보를 전달하고 있다면 alt를 비워둔다.
- **순수 장식 이미지는 alt=""(빈 문자열) 또는 `aria-hidden="true"`를 적용한다.** 스크린리더가 건너뛸 수 있어야 한다.

**Do/Don't 예시**

| Do | Don't | 이유 |
| --- | --- | --- |
| alt="김민준의 프로필 사진" | alt="사용자 프로필 이미지" | "이미지" 중복, 구체성 부족 |
| alt="SAZO 로고" | alt="로고 이미지" | "이미지" 중복 |
| alt="" (배경 패턴) | alt="배경 장식 이미지" | 장식 이미지는 빈 alt로 처리 |
| alt="일본 직구 상품 목록 배너" | alt="배너 링크 클릭" | "링크" 역할 중복, 내용을 설명해야 함 |

</Accordion>

<Accordion title="폼 라벨 및 유효성 검증">

**역할과 목표**
폼 라벨은 입력 필드의 정체를 밝히고, 유효성 검증 메시지는 올바른 입력 방법을 안내한다.

**글자 수 가이드**
- 폼 라벨: 2~10자 (명사/명사구)
- 유효성 검증 메시지: 10~30자 (1문장)

**톤 가이드**
- 폼 라벨: 명사형. 마침표 없음. 간결한 명사구.
- 유효성 검증: "~해 주세요" 패턴 기본. 사용자 실수를 비난하지 않으며, 올바른 입력 방법을 안내한다.
- 조건을 구체적으로 명시한다 (몇 자, 어떤 형식).
- **접근성:** 유효성 검증 메시지는 해당 입력 필드와 프로그래밍적으로 연결(`aria-describedby` 등)되어야 합니다. 텍스트만으로 오류 필드와 정정 방법을 완전히 식별할 수 있어야 하며, 색상이나 아이콘만으로 오류를 표시하지 않습니다.

**문장 구조 패턴**
- 폼 라벨: **[명사]** — "전화번호", "상세 주소", "받는 사람"
- 유효성 검증 형식 1: **[조건]으로 입력해 주세요** — "20자 이내로 입력해 주세요"
- 유효성 검증 형식 2: **[대상]을(를) 입력해 주세요** — "전화번호를 입력해 주세요"
- 유효성 검증 형식 3: **[구체적 형식 안내]** — "0으로 시작하는 올바른 전화번호를 입력하세요"

**Do/Don't 예시**

| Do | Don't | 이유 |
| --- | --- | --- |
| 3자 이상, 20자 이내로 입력해 주세요 | 3~20자 입력 필수 | 조건을 문장으로 풀어 쓰고, "필수"보다 "해 주세요"가 부드럽다 |
| 전화번호를 입력해 주세요 | 전화번호 미입력 | 해결 방법을 직접 안내한다 |
| 상세 주소 | 상세주소 | 띄어쓰기를 준수한다 |

**코드베이스 Before/After**

사례 1
- Before: `"요청사항을 입력 해주세요."` (apps/pc, translation.json)
- After: `"요청 사항을 입력해 주세요"` — "요청사항" → "요청 사항" 띄어쓰기, "입력 해주세요" → "입력해 주세요" 교정 (불필요한 공백 제거 + "해 주세요" 띄어쓰기), 마침표 생략.

사례 2
- Before: `"카드 명의인은 20자 이하로 입력해주세요."` (apps/pc, translation.json)
- After: `"카드 명의인은 20자 이내로 입력해 주세요"` — "이하" → "이내"로 통일(다른 검증 메시지와 일관성), "입력해주세요" → "입력해 주세요" 띄어쓰기.

**도움말 텍스트**
폼 라벨과 유효성 검증 사이에 위치하는 보조 안내 텍스트이다. 3가지 유형으로 구분한다.

| 유형 | 표시 시점 | 용도 | 예시 |
| --- | --- | --- | --- |
| 상시 표시형 | 항상 | 입력 형식, 조건 안내 | "영문, 숫자 포함 8자 이상" |
| 안심형 | 항상 | 개인정보 불안 해소 | "이 정보는 배송 목적으로만 사용됩니다" |
| 에러 예방형 | 입력 전 | 자주 틀리는 항목 사전 안내 | "하이픈(-) 없이 숫자만 입력해 주세요" |

**필수/선택 표시 기준**
- **필수 필드:** 라벨 뒤에 빨간 별표(*)를 붙인다. `aria-required="true"` 속성을 함께 적용한다.
- **대부분 필수인 폼:** 선택 필드에만 "(선택)" 텍스트를 붙인다. 필수가 기본이므로 별표를 생략하여 시각적 노이즈를 줄인다.
- **대부분 선택인 폼:** 필수 필드에만 "(필수)" 텍스트를 붙인다.

</Accordion>

**역할과 목표**
해당 섹션이나 페이지의 내용을 한눈에 파악하게 한다.

**글자 수 가이드**

**역할과 목표**
문제가 발생했음을 알리고, 사용자가 다음으로 취할 수 있는 행동을 제시한다.

- "오류"로 통일한다 ("에러" 사용 금지).
- 부정 표현을 최소화하고, 가능하면 해결 방법을 먼저 제시한다.
- 사용자 실수를 비난하는 어감을 피한다.
- **오류 발생 위치를 구체적으로 명시한다.** "결제 정보가 올바르지 않습니다"처럼 포괄적인 메시지 대신, 어떤 필드에서 문제가 발생했는지 특정한다. 스크린 리더 사용자는 화면 전체를 시각적으로 스캔할 수 없으므로, 위치 특정이 접근성 측면에서 특히 중요하다.
- **"유효하지 않은" 표현을 금지한다.** "유효하지 않은 이메일"은 개발자 관점의 표현이다. 구체적으로 무엇이 잘못되었는지 설명한다. (예: "이메일 형식을 확인해 주세요")
- **사용자 행동 오류는 능동태, 시스템/외부 원인은 수동태를 사용한다.** "이메일을 다시 확인해 주세요"(능동) vs "카드가 거절되었습니다"(수동).

**심각도별 표시 규칙**
오류의 심각도에 따라 표시 방식을 결정한다.

| 심각도 | 표시 방식 | 예시 | 자동 닫힘 |
| --- | --- | --- | --- |
| 경미 (입력 교정) | 인라인 텍스트 | 필드 아래 빨간 텍스트 | 수정 시 해제 |
| 보통 (작업 실패) | 토스트 | "URL을 복사하지 못했습니다" | 3~5초 |
| 심각 (흐름 차단) | 다이얼로그 | "결제를 처리할 수 없습니다. 다른 결제 수단을 선택해 주세요." | 수동 닫기 |
| 치명 (서비스 불가) | 전체 페이지 | "현재 서비스 점검 중입니다. 잠시 후 다시 방문해 주세요." | 해당 없음 |

**오류 표시의 3요소 조합 (시각 접근성)**
색상만으로 오류를 표시하면 색각 이상 사용자가 인지할 수 없다. 오류 상태는 반드시 다음 3가지를 조합한다.

| 요소 | 역할 | 예시 |
| --- | --- | --- |
| 색상 | 시각적 강조 (빨간색 테두리/텍스트) | `color-error` 토큰 |
| 아이콘 | 빠른 시각 식별 | 경고 아이콘, 에러 아이콘 |
| 텍스트 | 명확한 설명 | "전화번호를 입력해 주세요" |

- 텍스트는 **색상과 아이콘이 없어도 단독으로 이해 가능** 해야 한다.
- 나쁜 예: 필수 입력 필드를 빨간 테두리로만 표시 → 색각 이상 사용자는 어떤 필드에 문제가 있는지 알 수 없다.

| 라벨: "배송 알림" / 설명: "주문 상품의 배송 상태가 변경되면 알려드립니다" | 라벨: "알림" / 설명: 없음 | 어떤 알림인지, 언제 오는지 알 수 없다 |
| 라벨: "자동 로그인" / 설명: "다음에 앱을 열 때 자동으로 로그인됩니다" | 라벨: "자동 로드" | 기술 용어 대신 결과를 설명한다 |
| 라벨: "마케팅 수신 동의" / 설명: "프로모션, 이벤트 정보를 받습니다" | 라벨: "마케팅" / 설명: 없음 | 무엇에 동의하는지 명확해야 한다 |

</Accordion>

<Accordion title="권한 요청">

**역할과 목표**
카메라, 위치, 알림 등 시스템 권한을 요청할 때, 사용자가 "왜 필요한지"를 이해하고 수락할 수 있도록 맥락을 제공한다.

**글자 수 가이드**
- 사전 안내 타이틀: 5~15자
- 사전 안내 본문: 15~50자 (1~2문장)

**톤 가이드**
- **2단계 구조를 기본으로 한다:** (1) 앱 자체 화면에서 맥락 설명 → (2) 시스템 권한 프롬프트 표시. 시스템 프롬프트가 갑자기 뜨면 거부율이 높아진다.
- 권한이 필요한 이유를 사용자 **이득 관점** 에서 설명한다.
- "허용"과 "나중에" 옵션을 동등하게 제공한다. 거부 시 불이익을 암시하지 않는다.

**문장 구조 패턴**
- 사전 안내 형식: **[이득] + [권한 설명]** — "배송 상태를 실시간으로 받아보려면 알림을 허용해 주세요"

**Do/Don't 예시**

| Do | Don't | 이유 |
| --- | --- | --- |
| "배송 상태를 실시간으로 받아보려면 알림을 허용해 주세요" | (시스템 프롬프트 바로 표시) | 맥락 없이 권한을 요청하면 거부율이 높다 |
| 버튼: "알림 허용하기" / "나중에" | 버튼: "허용" / "거부" | "나중에"가 "거부"보다 부드럽다 |
| "사진을 첨부하려면 카메라 접근 권한이 필요합니다" | "카메라 권한을 허용해 주세요" | 왜 필요한지 설명해야 수락률이 올라간다 |

</Accordion>

</AccordionGroup>

## 콘텐츠 구조 접근성

<AccordionGroup>

<Accordion title="의미론적 헤딩 구조">

- `h1`은 페이지당 정확히 하나만 사용한다. 페이지의 주제를 대표하는 헤딩이다.
- 헤딩 계층을 건너뛰지 않는다. `h2` 없이 `h3`을 사용하면 스크린리더 사용자가 문서 구조를 파악할 수 없다.
- 헤딩을 텍스트 크기 조정 용도로 사용하지 않는다. 시각적 스타일은 CSS로, 의미 구조는 헤딩 태그로 분리한다.

</Accordion>

<Accordion title="테이블의 캡션과 헤더">

- `<caption>` 요소로 테이블 제목을 반드시 제공한다. 스크린리더가 테이블에 진입할 때 가장 먼저 읽어 주는 요소이다.
- `scope="col"` 또는 `scope="row"`로 헤더 범위를 명시한다. 스크린리더 사용자가 셀 간 이동할 때 해당 셀이 어떤 행/열에 속하는지 맥락을 유지할 수 있다.
- 복잡한 병합 셀 구조는 가능한 한 피하고, 단순한 행/열 구조를 권장한다.

</Accordion>

<Accordion title="읽기 순서와 DOM 순서">

- 시각적 배치 순서와 DOM(HTML) 순서를 일치시킨다. 스크린리더는 DOM 순서대로 콘텐츠를 읽기 때문에, CSS로 시각적 순서만 변경하면 정보 전달 순서가 어긋난다.
- 특히 모바일 반응형 레이아웃에서 데스크톱과 모바일의 DOM 순서가 달라지지 않도록 주의한다.

</Accordion>

</AccordionGroup>
